# Промпт для Claude Code: Веб-клиент электронной почты (ThunderWeb)

> Версия 2. Переработана под три приоритета: (1) много анимаций и красивый фронт,
> (2) быстрый бэкенд с мультиящиками, (3) понятное и кастомизируемое управление ящиками.

## 🎯 Цель проекта

Создай полноценный веб-клиент электронной почты **ThunderWeb** — быструю, красивую,
кастомизируемую альтернативу Thunderbird в браузере. С самого начала проект
рассчитан на **несколько почтовых ящиков одновременно** (мультиаккаунт + объединённый
входящий). Это базовое архитектурное требование, а не фича «на потом».

---

## ⚠️ Главные архитектурные принципы (прочитай первым)

1. **Мультиаккаунт — это фундамент.** Любой объект (письмо, папка, запрос) всегда
   привязан к `account_id`. Не проектируй ничего «на один ящик» с идеей расширить позже —
   переделывать дороже.
2. **Скорость = заголовки отдельно от тел + кэш + пул соединений.** Никогда не тяни тело
   письма ради списка. Список строится только из заголовков/конвертов, кэшированных локально.
3. **Соединение IMAP — НЕ потокобезопасно и НЕ переиспользуется для параллельных запросов.**
   Нужен пул соединений на каждый аккаунт (см. раздел «Производительность»).
4. **Кастомизация ящиков — отдельный слой состояния**, не зашитый в IMAP (порядок папок,
   цвета аккаунтов, закреплённые папки, объединённый инбокс — всё хранится локально у нас).

## 🏗️ Стек технологий

### Backend
- **Python 3.11+**, **FastAPI**, **uvicorn** (с `--workers` или gunicorn в проде)
- **IMAP-клиент — выбери осознанно (см. ниже).** Рекомендация: **`IMAPClient`** (синхронный,
  стабильный, поддерживает STARTTLS/XOAUTH2) **в пуле потоков через `run_in_executor`**.
  Это надёжнее, чем async-обёртки. Альтернатива — **`imap-tools`** (очень эргономичный, тоже sync).
  - ⚠️ **НЕ бери `aioimaplib` по умолчанию**: он слабо поддерживается, не имеет полноценной
    поддержки STARTTLS и XOAUTH2, что нам нужно. Используй его только если осознанно нужен
    нативный async IDLE и ты готов к его ограничениям.
- **`aiosmtplib`** — асинхронная отправка
- **Стандартный `email` + `email.header`** для парсинга MIME (надёжнее экзотики; добавь
  `mail-parser` только если столкнёшься со сложными структурами)
- **`aiosqlite` + SQLite** — локальный кэш метаданных писем и состояния аккаунтов **(критично для скорости!)**
- **`cryptography` (Fernet)** — шифрование сохранённых учёток ящиков на диске
  (НЕ `passlib` — пароли к IMAP нельзя хешировать, их надо уметь расшифровать для подключения)
- **`python-jose`** — JWT для сессии нашего приложения
- **`pydantic v2`**, **`python-dotenv`**

### Frontend
- **React 18 + TypeScript**, **Vite**, **Tailwind CSS v3**
- **Framer Motion** — анимации (основной инструмент)
- **`react-virtuoso`** (или `@tanstack/react-virtual`) — **виртуализация списка писем**.
  Обязательно: без неё длинные списки + анимации = тормоза.
- **`@dnd-kit/core` + `@dnd-kit/sortable`** — перетаскивание для кастомизации (порядок папок/аккаунтов)
- **Zustand** — клиентское состояние (UI, кастомизация, активный аккаунт)
- **TanStack Query v5** — серверные данные, кэш, фоновое обновление
- **React Router v6**, **Tiptap** (composer), **date-fns**, **lucide-react**
- **`DOMPurify`** — санитизация HTML писем, **`sonner`** (или react-hot-toast) — тосты

### Инфраструктура
- **Docker + docker-compose**; **nginx** как reverse proxy — опционально

## 🚀 Производительность бэкенда (приоритет №2 — раздел обязательный)

Цель: список писем открывается «мгновенно», переключение между ящиками — без ожидания,
объединённый инбокс из нескольких аккаунтов собирается быстро.

### 1. Пул соединений на каждый аккаунт
- На каждый `account_id` держи **пул из N открытых IMAP-соединений** (N=2–4), а не одно.
  IMAP-соединение нельзя использовать конкурентно — параллельные запросы берут разные соединения из пула.
- Учитывай лимиты сервера (например, Gmail — максимум 15 соединений на аккаунт). Делай пул настраиваемым.
- Реализуй **экспоненциальный backoff** и переподключение при `IMAP4.abort`/обрыве.
- Соединения в пуле «прогревай» при добавлении аккаунта, чтобы первый запрос не ждал хендшейк+логин.

### 2. Заголовки отдельно от тел (золотое правило)
- Для списка писем тяни **только конверт/заголовки**: `FETCH` с `ENVELOPE`, `FLAGS`,
  `BODYSTRUCTURE`, `INTERNALDATE` и при необходимости `BODY.PEEK[HEADER.FIELDS (...)]`.
- **Всегда `BODY.PEEK[...]`, а не `RFC822`** — чтобы не выставлять флаг `\Seen` при чтении.
- Тело письма тяни **лениво**, только когда пользователь открыл конкретное письмо.
- Превью текста (snippet) для списка генерируй из небольшого префикса `text/plain`-части, а не из всего тела.

### 3. Локальный кэш метаданных (SQLite) — главный ускоритель
- Таблица сообщений: `account_id, folder, uid, uidvalidity, envelope(json), flags, internaldate, snippet, has_attachments`.
- Список писем отдаём **сразу из SQLite**, а свежесть подтягиваем фоном (stale-while-revalidate).
- Синхронизация инкрементальная: запоминай `UIDVALIDITY` и `HIGHESTMODSEQ`/последний UID,
  тяни только новые/изменённые сообщения, а не всю папку заново.
- При смене `UIDVALIDITY` — инвалидируй кэш этой папки.

### 4. Пагинация и подгрузка
- Пагинация по **UID-окнам** (например, последние 50 UID, затем следующие 50), а не «скачать всё».
- Предзагрузка: при открытии папки префетчи тела первых 1–2 писем; на hover в списке —
  мягкий префетч тела (через TanStack Query `prefetchQuery`).

### 5. Объединённый входящий (Unified Inbox)
- Эндпоинт собирает заголовки из INBOX **всех аккаунтов параллельно** (asyncio.gather по аккаунтам),
  мёржит и сортирует по дате. Источник — кэш, не живой IMAP при каждом запросе.

### 6. Свежесть без сложного IDLE
- Для MVP — **лёгкий поллинг** (например, раз в 60 сек, настраивается) фоновой синхронизации новых писем.
- IMAP IDLE/real-time — вне MVP (можно добавить позже через отдельный воркер).

## 📂 Управление ящиками и кастомизация (приоритет №3 — раздел обязательный)

Это отдельный слой состояния, который хранится у нас (SQLite + Zustand на фронте), поверх IMAP.

### Управление аккаунтами (CRUD)
- Добавление ящика: хост/порт IMAP, хост/порт SMTP, логин, пароль, тип шифрования (SSL/TLS, STARTTLS).
- **Автоопределение настроек** по домену e-mail (таблица популярных провайдеров: Gmail, Яндекс,
  Mail.ru, Outlook, и т.д.) — пользователь вводит почту и пароль, остальное подставляется,
  с возможностью ручной правки.
- Редактирование, удаление, **включение/выключение** ящика (отключённый не синхронизируется).
- Проверка соединения кнопкой «Проверить» до сохранения.
- Учётки хранятся **зашифрованными (Fernet)**; ключ — из переменной окружения, не в коде.

### Кастомизация (всё сохраняется и переживает перезагрузку)
- **Цвет/акцент для каждого аккаунта** — цветная метка в сайдбаре и на письмах объединённого инбокса.
- **Перетаскивание (`@dnd-kit`)**: порядок аккаунтов и порядок папок внутри аккаунта.
- **Закрепление папок** (pin) наверх; **скрытие** ненужных системных папок.
- **Сворачиваемые группы** в сайдбаре: каждый аккаунт — сворачиваемая секция; плюс
  «Объединённый входящий» как отдельная секция сверху.
- Кастомные **имена-алиасы** для папок (отображаемое имя ≠ имя на сервере).
- Настройка интервала синхронизации и размера страницы списка — на аккаунт.

### Сайдбар (структура)
```
┌─────────────────────────┐
│ ✉  Написать             │
│─────────────────────────│
│ ★ Объединённый входящий  │  ← все аккаунты вместе
│─────────────────────────│
│ ▾ 🔵 Личная почта        │  ← аккаунт (цвет, сворачивается, drag)
│     📥 Входящие     12   │
│     📌 (закреплённые)    │
│     📤 Отправленные      │
│     🗑 Корзина           │
│ ▸ 🟢 Рабочая почта       │  ← свёрнут
│─────────────────────────│
│ ⚙ Аккаунты   🎨 Тема     │
└─────────────────────────┘
```

## 🎨 Дизайн, темы и анимации (приоритет №1)

### Визуальный стиль
Вдохновение — **Apple Mail**: чистый, премиальный, трёхколоночный (sidebar / список / просмотр).
Шрифты — системные SF Pro / `-apple-system` **во всём интерфейсе** (без моноширинного для тем
писем — это противоречит стилю Apple Mail). Чёткая типографическая иерархия: тема письма, превью,
мета-данные с правильным контрастом.

### Темы (4 штуки) + переключатель
- `light` (как macOS Apple Mail), `dark` (#1C1C1E), `nord` (холодная), `solarized` (тёплая).
- Реализация через CSS custom properties на `:root`, переключение мгновенное, выбор в localStorage.
- Учитывай цветные метки аккаунтов поверх любой темы (контраст должен сохраняться).

### Анимации — много, но не в ущерб скорости
Используем Framer Motion. Конкретные места:
- **Список писем:** появление со stagger — НО только для видимого окна виртуализации
  (`react-virtuoso` + анимация только видимых строк), иначе будет джанк на длинных списках.
- **Открытие письма:** slide/fade-in правой панели через `AnimatePresence`.
- **Переключение папок/аккаунтов:** crossfade содержимого списка.
- **Compose:** spring-анимация появления модалки, плавное разворачивание Cc/Bcc.
- **Сайдбар:** анимация сворачивания групп (height/opacity), плавный hover на пунктах.
- **Drag & drop:** анимированное переупорядочивание (`@dnd-kit` + layout-анимации Framer).
- **Микровзаимодействия:** hover на письмах, нажатия кнопок, появление/исчезновение тостов.
- **Layout-анимации** (`layout` prop) для плавных перестроений списка при удалении/перемещении письма.
- **Skeleton loading** при первой загрузке (но если есть кэш — показываем данные сразу, без скелетона).

### Обязательный «пол качества»
- **`prefers-reduced-motion`** — уважать: при включённом режиме отключать крупные движения,
  оставлять только мягкие фейды.
- Адаптив до мобильного (колонки сворачиваются в стек), видимый фокус с клавиатуры, доступность.
- Анимации на `transform`/`opacity` (GPU), без анимации `width/top/left` в горячих местах.

## ⚙️ Функциональность MVP

1. **Аккаунты:** добавить/изменить/удалить/выключить ящик, автоопределение настроек, проверка соединения.
2. **Авторизация в приложении:** JWT-сессия (httpOnly cookie, 24 ч) поверх нашего хранилища аккаунтов.
3. **Сайдбар:** объединённый инбокс + папки по каждому аккаунту, счётчики непрочитанного,
   кастомизация (drag, pin, скрытие, цвета, сворачивание).
4. **Список писем:** виртуализированный, из кэша, с анимациями; отметка прочитанного,
   обновление (poll + ручное).
5. **Просмотр письма:** HTML в изолированном `sandbox`-iframe; **блокировка внешних картинок
   по умолчанию** с кнопкой «Показать изображения» (приватность + скорость); fallback на plain text;
   вложения; действия (ответить/всем/переслать/удалить/в спам).
6. **Composer (Tiptap):** Кому (теги), Cc/Bcc, тема, форматирование, вложения (drag&drop),
   автосохранение черновика, выбор аккаунта-отправителя.
7. **Поиск:** сначала по локальному кэшу (мгновенно), при необходимости — IMAP SEARCH; debounce 400–500 мс;
   поиск в пределах аккаунта или по всем сразу.
8. **Вложения:** скачивание, inline-превью изображений, иконки по типу файла.

## 🔌 API Endpoints (Backend)

Замечание: имена папок содержат разделители и не-ASCII — передавай папку как
**query-параметр или в base64url**, не как сырой сегмент пути.

```
# Аккаунты
GET    /api/accounts                      # список ящиков (без паролей)
POST   /api/accounts                      # добавить ящик
PATCH  /api/accounts/{account_id}         # изменить (вкл/выкл, цвет, настройки)
DELETE /api/accounts/{account_id}         # удалить
POST   /api/accounts/test                 # проверить соединение до сохранения
GET    /api/accounts/autodetect?email=    # автоопределение IMAP/SMTP по домену

# Сессия приложения
POST   /api/auth/login                    # вход в приложение
POST   /api/auth/logout
GET    /api/auth/me

# Папки и кастомизация
GET    /api/accounts/{account_id}/folders
PATCH  /api/accounts/{account_id}/folders/layout   # порядок/pin/hide/алиасы

# Письма
GET    /api/accounts/{account_id}/messages?folder=&cursor=&limit=   # список (из кэша)
GET    /api/unified/messages?cursor=&limit=                         # объединённый инбокс
GET    /api/accounts/{account_id}/messages/{uid}?folder=            # одно письмо (тело лениво)
POST   /api/accounts/{account_id}/messages/{uid}/flags             # \Seen, \Flagged и т.п.
POST   /api/accounts/{account_id}/messages/{uid}/move?folder=
DELETE /api/accounts/{account_id}/messages/{uid}?folder=
POST   /api/accounts/{account_id}/sync                              # форсированная синхронизация

# Отправка, поиск, вложения
POST   /api/accounts/{account_id}/send
GET    /api/search?account_id=&q=&scope=                            # scope: account|all
GET    /api/accounts/{account_id}/messages/{uid}/attachments/{part}?folder=
```

## 🛡️ Технические требования

### Безопасность
- Учётки ящиков на диске — **только в зашифрованном виде (Fernet)**; ключ из env.
- HTML писем — `DOMPurify` + изолированный `sandbox` iframe; **внешние ресурсы блокируются по умолчанию**.
- CORS строго на домен фронта; вложения — только авторизованной сессии.

### Кодировки и MIME
- Заголовки — через `email.header.decode_header`; поддержка UTF-8, Windows-1251, KOI8-R в телах.
- Корректная обработка `multipart/mixed` и `multipart/alternative`, выбор `text/html` vs `text/plain`.
- **Обязательно протестировать на письмах с кириллицей в теме и теле.**

### Ошибки и устойчивость
- Все IMAP/SMTP ошибки → понятные HTTP-статусы; на фронте — тосты (не `alert`).
- Backoff и переподключение для нестабильных соединений; обработка протухания токенов (для OAuth позже).

## 💡 Советы и грабли
- **IMAP-соединение не потокобезопасно** — никогда не шарь одно соединение между конкурентными запросами. Пул!
- **Никогда не строй список писем из полных тел** — только заголовки/конверты из кэша.
- **`BODY.PEEK[]`, не `RFC822`** — иначе письма помечаются прочитанными при предпросмотре.
